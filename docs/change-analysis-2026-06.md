## 1. 가격·시간 구조 변경

### 요청
- 무료: 5분 (유지)
- **9,900원 → 15분** (현행과 동일: taste=9900/15분, 이미 일치)
- **14,900원 / 30분** (신규 티어)
- **30,000원 / 50분 (deep) 폐지**
- 마스터 채팅(10만/60분)·대면(20만/80분)·궁합 채팅(4,900/15분): 변경 없음

| 플랜키 | 현재 | 요청 | 조치 |
|--------|------|------|------|
| free | 0원 / 5분 | 0원 / 5분 | 유지 |
| taste | 9,900원 / 15분 | 9,900원 / 15분 | 유지 (이미 일치) |
| **(신규)** | — | **14,900원 / 30분** | **신규 플랜키 추가 필요** (예: `standard`) |

### 함께 손볼 곳 (시간 카운팅/표시)
- 세션 시간은 `PLAN_CONFIG.durationMinutes`로 계산되어 `expiresAt = startedAt + durationMinutes*60*1000`로 박힙니다 (routers.ts 330·384행, portoneRouter 55행, depositRouter 235행).
- 즉 **분(分) 값만 바꾸면 카운트다운·만료는 자동으로 따라옵니다.** 별도 타이머 로직 수정 불필요.
- 화면 카피(위 프런트 목록)는 자동이 아니라 **글자로 박혀 있어 수동 수정**이 필요합니다.

### ⚠️ deep(30,000/50분) 폐지 시 반드시 짚어야 할 것
1. **이미 deep로 결제·이용한 기존 기록**: DB의 `payments`/`consult_sessions`에 `planType="deep"`가 남아 있습니다. 폐지란 "신규 판매 중단"이지 "과거 기록 삭제"가 아니어야 합니다. → 라벨 표시는 살려두고, **구매 진입만 차단**하는 방식 권장. (enum에서 deep을 완전히 빼면 과거 기록 조회 시 라벨이 깨질 수 있음 → 라벨 맵은 보존)
2. **신규 플랜키 이름**: 14,900/30분을 어떤 키로 부를지 정해야 합니다(예: `standard`, `plus`, `main` 등). enum·스키마·4개 PLAN_CONFIG·화면 모두 같은 키를 써야 합니다.

### 확인 질문 (가격)
- (Q1) **14,900/30분 플랜의 키 이름**을 무엇으로 할까요? (deep을 폐지하지 않고 deep의 값만 30,000/50→14,900/30으로 "갈아끼우는" 방법도 있습니다. 이러면 신규 키가 필요 없고, 과거 deep 기록은 금액이 그대로 남아 안전합니다 — **이 방식이 가장 깔끔합니다.** 어떻게 할지 정해주세요.)
- (Q2) **event(0원/20분) 플랜**은 그대로 둘까요? (이벤트 시크릿 코드용으로 살아 있습니다.)
- (Q3) 가격 정의를 **단일 소스로 통합(권장안)** 할까요, 아니면 **네 곳 값만 맞추는 최소안**으로 갈까요?

## 2. AI 마무리 정리 행동 (신규 화법)

### 요청 개념
- 시간이 얼마 남지 않으면 "오늘 나눈 것을 정리하면…" 식으로 갈무리.
- 마스터 추가 지시: **"상담이 끝나도 '내 상담실'에서 PDF로 다운로드 받으실 수 있다"** 는 안내를 곁들이면 더 좋다.

### 들어갈 위치 (제안)
- `server/masterPromptV4.ts`의 **■ 화법 원칙** 단락(305행 근처) 또는 그 아래 **첫 인사/현실 연동 원칙**(335행) 뒤에 새 소항목으로.
- 사실 관계 확인됨: PDF 다운로드 기능은 **이미 존재**합니다. `server/consultPdf.ts`, 테스트 `consult.downloadPdf.test.ts`, CS 안내문(cs.ts 84행 "내 상담실 탭에서 확인")까지 실재. 따라서 "PDF로 받으실 수 있다"는 안내는 **허위가 아니라 실제 기능과 일치** → 안전하게 멘트 가능.

### 충돌·중복 점검
- **충돌 없음.** 다만 두 가지 인접 규칙과 "톤"을 맞춰야 함:
  - (a) "다섯. 반드시 강점과 희망으로 마무리한다"(100행) — 기존 마무리 원칙. 신규 "시간 갈무리"는 이와 **상충하지 않고 보완**(희망으로 맺되, 끝에 정리+PDF 안내). 단, 둘이 따로 놀지 않게 "희망으로 맺으면서 갈무리"로 엮는 문장이 좋음.
  - (b) "여백의 미학 — 다 말하지 않는다"(309행) — 갈무리가 너무 길어지면 이 원칙과 부딪힘. → 갈무리는 **짧게**.
- ⚠️ **기술적 주의:** "시간이 얼마 남지 않으면"을 **프롬프트가 스스로 알 수 있는가?** AI는 현재 세션의 잔여 시간을 모릅니다. 두 가지 길:
  - (i) **순수 프롬프트 방식**: "대화가 마무리 국면에 접어들면" 정도의 **맥락 기반**(시간 수치 없이). → 새 기술 불필요, 단 "정확히 몇 분 남음"은 모름.
  - (ii) **잔여 시간 주입 방식**: 서버가 매 메시지에 "잔여 N분"을 프롬프트에 넣어줌. → 정확하지만 가벼운 코드 추가 필요.
  - 클로드 정리본은 "시간 기반 기능 불필요"라 했으므로 기본은 **(i)** 로 잡되, 원하시면 (ii)도 가능. **확인 필요.**

### 확인 질문 (마무리)
- (Q4) 갈무리 트리거를 (i) 맥락 기반(시간 수치 없음) / (ii) 잔여 시간 주입 중 무엇으로?
- 실제 문장은 클로드가 빚어서 전달 예정 — 위치는 **화법 원칙 단락 하단**으로 확정 제안.

## 3. 능동적 제안 행동 (능동적 드리블 확장)

### 요청 개념
- 사용자 입력이 짧고 모호할 때("음… 잘 모르겠어요") AI가 "다른 분들이 많이 여쭤보신 건…" 식으로 먼저 제안.
- **시간 기반 감지 불필요**, 입력 내용 기반으로 프롬프트 수준만.

### 들어갈 위치 (제안) — "능동적 드리블" 단락은 **세 군데**에 분산되어 있음
1. `server/masterPromptV4.ts` 105–114행 **■ 능동적 드리블** (첫 답변 짧게/범위 좁히기) — *주 위치*
2. `server/masterPromptV4.ts` 117–125행 **■ 되치기 분기 설계** (되묻기 전 답 준비)
3. `server/masterPromptV4.ts` 313행 **화법 원칙 내 능동적 드리블** (숫자·결과 되묻기)
4. `server/promptLayers.ts` 144행 (동일 취지의 레이어판)

→ 신규 "짧고 모호할 때 먼저 제안"은 **1번(■ 능동적 드리블) 단락 안의 새 소항목**이 가장 자연스럽습니다. 현재 1번은 "고객이 *구체적* 상황을 물을 때"를 다루는데, 신규 건은 정반대인 "**고객이 막막해할 때**"라 짝을 이루는 보완 규칙이 됩니다.

### 충돌·중복 점검
- **부분 중복 주의:** "되묻기"가 이미 강조되어 있어, 자칫 AI가 **매번 되묻기만** 하고 답을 안 줄 위험. 신규 규칙은 "되묻되, 모호하면 **선택지(예시 질문)를 먼저 제시**"로 방향이 다르므로, "빈 되묻기 금지(120행)"와 정합. → 충돌 아니라 보완. 단 문장에서 "되묻기 대신 제안"이 아니라 "막막해하면 제안으로 길을 연다"로 구분해 적어야 함.
- promptLayers.ts와 masterPromptV4.ts **양쪽에 같은 취지가 살아 있어**, 한쪽만 고치면 경로(개인상담 vs 궁합 등)에 따라 행동이 달라질 수 있음 → **어느 경로에 적용할지** 확인 필요.

### 확인 질문 (능동적 제안)
- (Q5) 이 규칙을 개인상담(masterPromptV4)만 적용? 아니면 궁합 등 레이어(promptLayers)에도?
- 실제 문장은 클로드가 빚어서 전달 — 위치는 **■ 능동적 드리블 단락 하단**으로 확정 제안.

## 4. 실전 질문 패턴 뱅크 (RAG) — 보류

- 지시대로 **이번에 작업하지 않음.** 콘텐츠 준비 후 별도 진행.
- 마스터 메모("마누스가 직접 해줄 수도 있을 듯하나 지저분해지고 AI 역할에 혼란 줄까 고민")에 대한 의견만 남깁니다:
  - RAG에 "질문 예시"를 많이 넣으면, AI가 **마스터의 실제 화법을 흉내내는 데는 도움**되지만, 잘못하면 **앵무새처럼 예시 질문을 그대로 따라 읽는** 부작용이 생길 수 있습니다. 그래서 "혼란 줄까" 하신 직감이 타당합니다.
  - 권장: 질문 문장을 **그대로 검색돼 나오는 형태**로 넣지 말고, "이런 국면에선 이런 각도로 파고든다"는 **관법(접근 방식) 노트** 형태로 가공해 넣는 게 깔끔합니다. 콘텐츠 주시면 그 가공까지 제가 도울 수 있습니다.

## 5. 가격과 무관하나 즉시 보고할 잔존 버그 (지금 고치지 않음)

`server/masterPromptV4.ts` **348행**:
```
현재 연도: 2026년 병오년(丙午年) — 자체 계산 금지, 이 값만 사용.
```
- 이건 우리가 막 끝낸 **입춘 보정 시점 수술과 충돌**합니다. 해가 2027년으로 바뀌어도 이 줄이 "지금은 2026년"이라고 못 박아, 애써 만든 동적 시점 로직을 덮어쓸 위험이 있습니다.
- 바로 아래 357~369행에는 이미 "표 그대로 인용/입춘 세수/시점 애매하면 확인"이 잘 들어가 있어, **348행만 동적 표현으로 바꾸거나 삭제**하면 됩니다(예: "현재 연도·간지는 【현재 시점】 섹션의 코드 확정값을 따른다").
- 가격 작업과 별개의 작은 수정이라, 1번 진행하실 때 같이 처리할지만 알려주시면 됩니다.
